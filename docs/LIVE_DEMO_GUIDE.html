Logout and invalidate token

Expected: Logout returns 200 "Logged out successfully". Second request returns 401 — token is blacklisted via JTI
Viva Evidence: JWT Security
  • 7-step validation: Signature (HMAC-SHA256) → Expiry → Issuer → JTI blacklist → Subject → Role → SecurityContext
  • Account lockout: 5 failed attempts → 15-minute cooldown (ConcurrentHashMap tracking)
  • Password policy: @Pattern regex enforcing uppercase + lowercase + digit + special char
  • BCrypt cost=12: ~100ms per hash, immune to rainbow tables, constant-time comparison

6.3

Exhaust rate limit (send 101 requests)

Note: This takes ~1-2 minutes. The 101st request will be rejected.

Run (use testuser to avoid locking admin)TEST_TOKEN=$(curl -s -X POST http://localhost:8100/auth/login \ -H "Content-Type: application/json" \ -d '{"username":"testuser","password":"User@123"}' | jq -r '.token') for i in {1..101}; do CODE=$(curl -s -o /dev/null -w "%{http_code}" -X POST http://localhost:8100/api/ask \ -H "Content-Type: application/json" \ -H "Authorization: Bearer $TEST_TOKEN" \ -d '{"prompt":"test","useReActAgent":false}') [ "$CODE" = "429" ] && echo "Request $i: $CODE (RATE LIMITED!)" && break [ $((i % 20)) -eq 0 ] && echo "Request $i: $CODE (OK)" done
Expected: Requests 1-100 return 200, request 101 returns 429 Too Many Requests with Retry-After header
Viva Evidence: Rate Limiting Design

"We use Bucket4j token-bucket algorithm — each user gets 100 tokens, refilled every 60 minutes. It's per-user, not global, so one heavy user can't block everyone. The algorithm is O(1) for both check and consume operations. Headers tell the client exactly how many requests remain."

Viva Questions & Answers

10 Q&As
Q: "Why 3 layers specifically?"
"Defence-in-depth from military strategy. Each layer catches a different threat class: structural patterns (L1), semantic intent (L2), data leakage (L3). If one fails, others still protect. These map to OWASP LLM Top 10 risk categories LLM01, LLM02, and LLM06."
Q: "How is this different from OpenAI's moderation?"
"Three differences: (1) 100% on-premise, GDPR-compliant. (2) 3 independent layers vs single-layer. (3) Colang DSL enables custom domain-specific policies."
Q: "What if a guardrail service goes down?"
"Fail-closed. Unreachable service = request blocked. Resilience4j circuit breakers prevent cascading failures (50% threshold, 10-request window)."
Q: "Don't 3 layers make it slow?"
"Parallel execution via Mono.zip(). Total ~90ms (slowest layer), not 130ms (sum). That's 5.6% of 1.6s total. LLM inference is the bottleneck."
Q: "What is Colang?"
"NVIDIA's DSL for declarative guardrail rules. Security teams write patterns without code changes or redeployment."
Q: "How did you test?"
"206 test methods, 29 classes. Unit (Mockito), integration (Spring Boot Test + H2), smoke, and performance tests. JaCoCo enforces 80% line / 70% branch coverage."
Q: "What is OWASP LLM Top 10?"
"10 most critical LLM security risks. LLM01 = Prompt Injection, LLM02 = Insecure Output, LLM06 = PII Disclosure. We cover 9 of 10."
Q: "Explain your CI/CD pipeline."
"15 stages, 7 blocking gates: unit tests (80% coverage), SpotBugs (0 issues), SonarQube quality gate, OWASP CVE scan (CVSS<7), Trivy container scan, Karate E2E tests. Shift-left security."
Q: "What about citations in your paper?"
"20 IEEE references across 5 categories. Greshake et al. motivated Layer 1, LlamaGuard paper [3] motivated Layer 2, Presidio [5] motivated Layer 3. NIST SSDF [6] informed our pipeline design."
Q: "What would you improve?"
"5 things: DAST via OWASP ZAP, Redis-backed JWT blacklist + rate limiter for distributed deployment, adversarial red-team testing, Istio mTLS, AI drift monitoring per NIST AI RMF."
