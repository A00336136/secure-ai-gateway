Preface

Why This Book Exists

The proliferation of Large Language Models across enterprise systems has outpaced the development of adequate security frameworks. This guide bridges that gap.

The year 2025 marked an inflection point for AI deployment in enterprise environments. Organizations rushed to integrate Large Language Models into their workflows, often prioritizing capability over security. The consequences were predictable: prompt injection vulnerabilities exposed confidential data, adversarial inputs bypassed safety mechanisms, and the absence of audit trails made regulatory compliance virtually impossible.

The Secure AI Gateway was conceived as a comprehensive response to these challenges. Drawing from established principles—defense-in-depth, fail-closed architectures, zero-trust networking—and combining them with AI safety frameworks from NVIDIA, Meta, and Microsoft, we engineered a system that demonstrates enterprise-grade AI security is both achievable and practical.

This guide is structured as a progressive journey. Parts I–II establish the foundational architecture and security mechanisms. Parts III–IV examine AI-specific components: guardrails, inference, and observability. Parts V–VII cover API design, containerized deployment, CI/CD automation, and testing methodology.

About the Technology Stack

This project leverages Spring Boot 3.4.3 on Java 21 LTS, Ollama for local LLM inference, NVIDIA NeMo Guardrails v0.10.0, Meta LlamaGuard 3, Microsoft Presidio v2.2, and Bucket4j—all containerized via Docker Compose with Jenkins CI/CD and SonarQube quality gates.

Chapter 1

Introduction to the Secure AI Gateway

Large Language Models represent one of the most consequential technological advances of the decade. Their deployment without adequate security infrastructure represents one of the most consequential risks.

1.1 The Threat Landscape for AI Systems

The integration of Large Language Models (LLMs) into enterprise workflows has introduced a category of security vulnerabilities that traditional application security frameworks were never designed to address. Unlike conventional software systems, LLM-based systems accept free-form natural language—an attack surface of virtually unlimited dimensionality.

The OWASP Top 10 for LLM Applications (2025 revision) identifies the following critical threat vectors:

LLM01: Prompt Injection
Adversarial inputs that manipulate the model into ignoring system instructions, revealing confidential prompts, or generating prohibited content. Both direct injection (user-crafted) and indirect injection (embedded in retrieved documents) pose significant risks.
LLM02: Insecure Output Handling
LLM-generated content consumed by downstream systems without validation, enabling XSS, SSRF, or code injection when model outputs are rendered or executed.
LLM06: Sensitive Information Disclosure
Models reproducing personally identifiable information (PII) in responses, violating GDPR Article 17 and CCPA Section 1798.100.
LLM10: Unbounded Consumption
Absence of rate limiting enabling resource exhaustion attacks through sustained request volume.

The Secure AI Gateway addresses each threat vector through a defense-in-depth architecture layering multiple, independent security mechanisms in a fail-closed configuration.

JWT Authentication and Cryptographic Security

JSON Web Tokens provide the stateless authentication foundation that enables horizontal scaling without shared session infrastructure.

4.1 JSON Web Token Specification (RFC 7519)

A JWT is a compact, URL-safe means of representing claims transferred between two parties. Each token consists of three Base64URL-encoded segments:

Header
{"alg":"HS384"}
Payload (Claims)
sub, role, jti, iss, iat, exp
Signature
HMAC-SHA384(header.payload, secret)
// Decoded JWT Payload
{
  "sub": "testuser",                    // Subject (username)
  "role": "USER",                       // Authorization role
  "jti": "8a4789ae-6c84-4890-...",      // Unique JWT ID (UUID)
  "iss": "secure-ai-gateway",            // Issuer identity
  "iat": 1774226249,                    // Issued at (Unix epoch)
  "exp": 1774229849                     // Expires (iat + 3600s)
}

4.2 HMAC-SHA384 Signing and Key Derivation

The gateway employs HMAC-SHA384 producing a 384-bit message authentication code, providing 192 bits of security strength—well above the NIST SP 800-107 minimum of 128 bits.

4.3 Token Generation (JwtUtil.java)

public String generateToken(String username, String role) {
    return Jwts.builder()
        .subject(username)
        .claim("role", role)
        .id(UUID.randomUUID().toString())     // JTI for replay prevention
        .issuer("secure-ai-gateway")
        .issuedAt(new Date())
        .expiration(new Date(System.currentTimeMillis() + expirationMs))
        .signWith(signingKey)                  // HMAC-SHA384
        .compact();
}

4.5 JwtAuthenticationFilter.java

The filter extends OncePerRequestFilter, ensuring exactly one execution per request:

@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain chain) {

        String header = request.getHeader("Authorization");
        if (header != null && header.startsWith("Bearer ")) {
            String token = header.substring(7);
            if (jwtUtil.validateToken(token)) {
                String username = jwtUtil.getUsername(token);
                String role = jwtUtil.getRole(token);

                // Create Spring Security authentication
                var auth = new UsernamePasswordAuthenticationToken(
                    username, null,
                    List.of(new SimpleGrantedAuthority(
                        "ROLE_" + role.toUpperCase(Locale.ROOT))));

                SecurityContextHolder.getContext()
                    .setAuthentication(auth);
            }
        }
        chain.doFilter(request, response);
    }
}

4.6 BCrypt Password Hashing (Cost Factor 12)

User passwords are stored using BCrypt with cost factor 12 (212 = 4,096 iterations). At this cost, a single hash takes ~250ms—imperceptible to users but requiring ~3 days to test one million password candidates.

@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12);
}

Key Takeaways
  • HMAC-SHA384 provides 192-bit security strength, exceeding NIST minimums
  • Seven-step validation catches forgery, replay, expiry, and privilege escalation
  • JTI (JWT ID) provides replay prevention via ConcurrentHashSet blacklist
  • BCrypt cost 12 requires ~3 days per million password guesses
  • JWT_SECRET has no default—app fails to start without it (Zero Trust)

Spring Security Filter Chain Configuration

Spring Security 6.4 provides the comprehensive security framework that orchestrates authentication, authorization, CSRF protection, CORS policy, and HTTP security headers.

5.1 SecurityConfig.java Explained

The security configuration is centralized in a single @Configuration class that defines the entire SecurityFilterChain:

@Configuration
@EnableWebSecurity
@EnableMethodSecurity(prePostEnabled = true)
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(
            HttpSecurity http,
            JwtAuthenticationFilter jwtFilter) throws Exception {
        http
            .csrf(csrf -> csrf
                .csrfTokenRepository(CookieCsrfTokenRepository
                    .withHttpOnlyFalse())
                .ignoringRequestMatchers(
                    "/auth/**", "/actuator/**",
                    "/h2-console/**", "/api/**"))
            .sessionManagement(sm ->
                sm.sessionCreationPolicy(
                    SessionCreationPolicy.STATELESS))
            .cors(cors -> cors.configurationSource(
                corsConfigurationSource()))
            .headers(headers -> headers
                .frameOptions(fo -> fo.sameOrigin())
                .xssProtection(XXssConfig::disable)
                .contentSecurityPolicy(csp -> csp
                    .policyDirectives("default-src 'self'..."))
                .httpStrictTransportSecurity(hsts ->
                    hsts.includeSubDomains(true)
                        .maxAgeInSeconds(31536000))
                .referrerPolicy(rp -> rp.policy(
                    ReferrerPolicy.STRICT_ORIGIN_WHEN_CROSS_ORIGIN)))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/auth/**").permitAll()
                .requestMatchers("/actuator/health").permitAll()
                .requestMatchers("/swagger-ui/**").permitAll()
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated())
            .addFilterBefore(jwtFilter,
                UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }
}

5.2 CSRF Protection Strategy

CSRF (Cross-Site Request Forgery) protection is disabled for API endpoints (/api/**, /auth/**) because these use JWT Bearer tokens in the Authorization header, which are inherently immune to CSRF attacks (browsers never automatically attach Authorization headers). For any future browser-rendered forms, Cookie-based CSRF tokens remain active.

Chapter 6

PII Detection and Redaction Engine

Personally Identifiable Information leakage through AI systems represents one of the most significant compliance risks. The PII Redaction Engine provides dual-pass protection on both inputs and outputs.

6.1 What is PII?

PII encompasses any information that can directly or indirectly identify an individual. Under GDPR Article 4, this includes names, identification numbers, location data, email addresses, and online identifiers. In AI systems, PII can leak through model responses that reproduce training data or echo back user-provided sensitive information.

6.3 PiiRedactionService.java Architecture

@Service
public class PiiRedactionService {

    public String redact(String text) {
        // Apply all 12 patterns sequentially
        for (RedactionRule rule : rules) {
            text = rule.pattern().matcher(text)
                .replaceAll(rule.replacement());
        }
        return text;
    }

    public boolean containsPii(String text) {
        return rules.stream()
            .anyMatch(r -> r.pattern().matcher(text).find());
    }

    public Set<String> detectPiiTypes(String text) {
        return rules.stream()
            .filter(r -> r.pattern().matcher(text).find())
            .map(RedactionRule::name)
            .collect(Collectors.toSet());
    }
}

6.4 Dual-Pass Redaction

PII redaction occurs at two points in the pipeline:

  1. Input pass (Stage 3): Guardrails check if the user's prompt contains PII
  2. Output pass (Stage 5): The LLM's response is scanned and redacted before returning to the client

This dual-pass approach ensures PII is caught regardless of whether it was provided by the user or generated by the model.

6.5 Credit Card Validation (Luhn Algorithm)

Credit card numbers are validated using the Luhn algorithm (ISO/IEC 7812-1) to avoid false positives on random 16-digit sequences. The algorithm doubles every second digit from the right, sums the digits, and checks if the total is divisible by 10.

Key Takeaways
  • 12 regex patterns cover email, SSN, credit cards, IBAN, phone numbers (IE/INTL/US), DOB, passport, IPv4, and IPv6
  • Dual-pass redaction protects both input prompts and output responses
  • Credit card validation uses Luhn algorithm to minimize false positives
  • Configurable via pii.redaction.enabled property

Ollama Integration — Local LLM Inference

Ollama enables local Large Language Model inference, ensuring complete data sovereignty—no prompts or responses ever leave the organization's infrastructure.

7.1 Why Local LLMs?

Cloud-based LLM APIs (OpenAI, Anthropic, Google) require transmitting potentially sensitive data to external servers. In regulated industries (healthcare, finance, government), this creates compliance obstacles under GDPR, HIPAA, and SOX. Ollama solves this by running models entirely on local hardware.

7.2 OllamaClient.java — REST API Integration

@Service
public class OllamaClient {

    public String generateResponse(String prompt) {
        Map<String, Object> body = Map.of(
            "model", model,              // llama3.1:8b

            "stream", false,

        // GET /api/tags - returns 2xx if Ollama is running

            String.class).getStatusCode().is2xxSuccessful();
    }
}

8.2 GuardrailsOrchestrator.java — Parallel Evaluation

@Service
public class GuardrailsOrchestrator {

    public GuardrailsEvaluation evaluate(String prompt) {
        // Parallel evaluation via Project Reactor
        return Mono.zip(
            nemoClient.evaluate(prompt)
                .subscribeOn(Schedulers.boundedElastic()),
            llamaGuardClient.evaluate(prompt)
                .subscribeOn(Schedulers.boundedElastic()),
            presidioClient.evaluate(prompt)
                .subscribeOn(Schedulers.boundedElastic())
        ).map(tuple -> {
            // FAIL-CLOSED: any blocked = all blocked
            boolean blocked = tuple.getT1().blocked()
                || tuple.getT2().blocked()
                || tuple.getT3().blocked();
            return new GuardrailsEvaluation(blocked, ...);
        }).block();
    }
}

Performance: Parallel evaluation completes in ~90ms vs ~160ms sequential (44% improvement).

Chapter 9

ReAct Agent — Multi-Step Reasoning

The ReAct (Reasoning + Acting) framework enables the LLM to decompose complex queries into iterative reasoning steps and tool-use actions.

9.1 What is ReAct?

Proposed by Yao et al. (2022), ReAct interleaves chain-of-thought reasoning with concrete actions, grounding the model's reasoning in observable outcomes. Each iteration follows a Thought → Action → Observation loop.

9.2 Implementation (ReActAgentService.java)

@Service
public class ReActAgentService {

    private static final int MAX_STEPS = 10;

    public AgentResult execute(String userPrompt) {
        List<AgentStep> steps = new ArrayList<>();
        StringBuilder scratchpad = new StringBuilder();

        for (int i = 0; i < MAX_STEPS; i++) {
            // Ask LLM to produce next step
            String llmOutput = ollamaClient.generateResponse(
                buildReActPrompt(userPrompt, scratchpad));

            // Parse: Thought, Action, Action Input
            String thought = parseThought(llmOutput);
            String action  = parseAction(llmOutput);
            String input   = parseActionInput(llmOutput);

            if ("answer".equals(action)) {
                return new AgentResult(parseFinalAnswer(llmOutput), steps);
            }

            // Execute tool and get observation
            String observation = executeTool(action, input);
            steps.add(new AgentStep(thought, action, input, observation));
            scratchpad.append(formatStep(steps.getLast()));
        }
        return new AgentResult("Max steps reached", steps);
    }
}

9.4 Safety Limits

Maximum 10 iterations prevents infinite loops. If the agent cannot reach an answer within 10 steps, it returns partial results with a fallback message.

Chapter 10

Rate Limiting with Bucket4j

The token bucket algorithm provides fair, per-user rate limiting that prevents resource exhaustion while accommodating legitimate burst traffic.

10.1 Token Bucket Algorithm

Each user receives a virtual "bucket" containing 100 tokens. Each API request consumes one token. Tokens refill at a rate of 100 per 60 minutes (greedy refill). When the bucket is empty, requests return HTTP 429 with a Retry-After header.

10.2 Implementation

@Service
public class RateLimiterService {

    private final Map<String, Bucket> userBuckets = new ConcurrentHashMap<>();

    public boolean tryConsume(String username) {
        Bucket bucket = userBuckets.computeIfAbsent(username, this::createBucket);
        return bucket.tryConsume(1);
    }

    public long getRemainingTokens(String username) {
        return userBuckets.getOrDefault(username, createBucket(username))
            .getAvailableTokens();
    }

    public void resetBucket(String username) {
        userBuckets.remove(username);  // Admin operation
    }

    private Bucket createBucket(String key) {
        return Bucket.builder()
            .addLimit(Bandwidth.classic(100,
                Refill.greedy(100, Duration.ofMinutes(60))))
            .build();
    }
}
Key Takeaways
  • 100 tokens/hour per user with greedy refill
  • ConcurrentHashMap provides thread-safe per-user isolation
  • Admin endpoint DELETE /admin/rate-limit/{username} resets buckets
  • For multi-node: replace with Bucket4j Redis backend (config-only change)

JSON Web Token (RFC 7519); compact, URL-safe token for transmitting authentication claims.
JTI
JWT ID; unique identifier enabling replay prevention via blacklist tracking.
LLM
Large Language Model; neural network trained on text corpora for natural language tasks.
Luhn Algorithm
Checksum formula (ISO 7812-1) for validating credit card numbers.
NER
Named Entity Recognition; NLP technique for identifying entities (persons, organizations) in text.
PII
Personally Identifiable Information; data that can identify a specific individual.
ReAct
Reasoning + Acting; paradigm where LLM alternates between reasoning and tool-use actions.
Token Bucket
Rate limiting algorithm using a virtual bucket of tokens consumed per request and refilled over time.
Zero Trust
Security model requiring explicit verification for every request with no implicit trust.
