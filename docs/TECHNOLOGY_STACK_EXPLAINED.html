The Big Picture — What Is This Project?

Understanding the entire system in 60 seconds

▼

✈ Think of an Airport Security System.

Passengers (users) want to board a plane (use AI). But before they reach the plane, they must pass through three security checkpoints that all run at the same time:

🛂 Checkpoint 1 (NeMo): The ID checker — compares your name against a banned list. Fast but can be fooled by a fake ID.

📷 Checkpoint 2 (LlamaGuard): The X-ray machine — looks INSIDE your bags and understands what's dangerous, even if it's hidden cleverly.

🔍 Checkpoint 3 (Presidio): The metal detector — beeps when it finds sensitive items like credit cards, passports, or personal documents.

If ANY checkpoint says STOP, you don't board. The plane (AI) never sees dangerous passengers.

That's SecureAI Gateway. But for AI prompts, not airport passengers.

This project is NOT an AI model. It is a security gateway — like a firewall, but for AI. The AI model (LLaMA) is a pluggable component. You can swap it out by changing one setting, just like changing the engine in a car without rebuilding the car.

What it IS

✅ A security layer between users and AI
✅ 3 guardrail checkpoints in parallel
✅ 100% on-premise (no cloud)
✅ Model-agnostic (swap any AI model)
✅ Open-source on GitHub

What it is NOT

❌ NOT an AI chatbot
❌ NOT a model training tool
❌ NOT tied to any AI vendor
❌ NOT cloud-dependent
❌ NOT a simple API wrapper

The Novel Contribution

No published work combines NeMo + LlamaGuard + Presidio in parallel. This is the first documented 3-layer parallel guardrail architecture for LLMs, with fail-closed security and on-premise deployment.

Complete Request Flow — What Happens When You Send a Prompt

Step-by-step journey of every user request through the system

▼
1
User sends a prompt

Example: "What is the capital of Ireland?" via POST /api/ask with a JWT Bearer token in the Authorization header.

2
JWT Authentication Filter (~2ms)

The system checks: Is this user logged in? Is the token valid? Has it expired? Is it on the blacklist (logged out)? Uses HMAC-SHA256 signature verification — 7 checks in total. If any fails → 401 Unauthorized.

3
Rate Limiter Check (~1ms)

Each user has a "bucket" of 100 tokens. Every request consumes 1 token. When the bucket is empty (100 requests used in 1 hour) → 429 Too Many Requests with Retry-After: 3600 header. The bucket refills automatically every hour.

4
Three Guardrail Layers — ALL AT THE SAME TIME (~300ms total)

This is the key innovation. Using Mono.zip() from Project Reactor, all three checkpoints evaluate the prompt simultaneously on separate threads:

🛡 Layer 1 (NeMo): Pattern matching — checks against 97 known attack patterns (jailbreak + harmful content). ~40ms.

🕵 Layer 2 (LlamaGuard): AI understanding — classifies the prompt against 12 safety categories (S1-S12). ~200ms.

🔍 Layer 3 (Presidio): PII detection — scans for credit cards, SSNs, emails, phone numbers. ~15ms.

5
Decision Gate — Fail-Closed Logic

If ANY of the 3 layers says "block", the entire request is denied → 422 Unprocessable Entity. The AI model never sees the prompt. This is called "fail-closed" because when in doubt, we block rather than allow.

6
AI Inference via Ollama (~1-3 seconds)

Only if ALL 3 layers pass does the prompt reach the AI model (LLaMA 3.1 8B running on your M3 Max GPU via Apple MLX). The model generates a response. If useReActAgent=true, the ReAct multi-step reasoning agent handles it instead (more on this later).

7
PII Response Redaction

Even after the AI generates a response, we scan the OUTPUT for any leaked personal data. If the AI accidentally mentions a credit card number in its response, it gets redacted before reaching the user.

8
Audit Logging + Response

Every request is logged to PostgreSQL (username, prompt, response, PII detected?, blocked?, duration, IP address). The response is returned to the user as 200 OK with the AI-generated text.

Prove it on localhost:3333: Go to Interactive Tester tab → Login as Admin → Click "Safe Prompt" → Watch the Request Pipeline panel on the left show each step turning green (PASS) in real-time, ending with 200 OK. Then click "Harmful Content" and watch steps turn red (BLOCKED) at 422.

Rate Limiting — How It Works In Detail

The traffic cop that prevents any single user from overloading the system

BUILT-IN ▼

🚰 Think of a water tank with a tap.

Each user gets their own water tank (bucket) that holds 100 litres (tokens). Every time you make a request, 1 litre drains out. A pipe slowly refills the tank — 100 litres every hour. When the tank is empty, the tap stops working and you see "429 Too Many Requests — come back when the tank refills."

This means:

• Burst: You can send 100 requests very quickly (the tank is full)
• Sustained: After that, you're limited to the refill rate
• Per-user: User A's bucket is separate from User B's — one heavy user can't block others
• Automatic: The bucket refills on its own — no admin intervention needed

The Token Bucket Algorithm (Bucket4j 8.10.1):

  1. When a user is first seen, a new bucket is created with 100 tokens
  2. Each API request to /api/ask calls rateLimiterService.tryConsume(username)
  3. If the bucket has tokens ≥ 1, one token is consumed and the request proceeds
  4. If the bucket has 0 tokens, the request is immediately rejected with HTTP 429
  5. The bucket refills at a rate of 100 tokens per 60 minutes (greedy refill)
  6. Every response includes headers: X-Rate-Limit-Remaining: 97 and X-Rate-Limit-Capacity: 100
  7. When rejected, the response includes: Retry-After: 3600 (seconds until reset)

How to prove it in the demo:

1. Send a request on localhost:3333 → check the API Response for X-Rate-Limit-Remaining header (should decrease with each request)

2. To demonstrate the limit being hit, use the terminal:

for i in {1..101}; do curl -s -o /dev/null -w "Request $i: %{http_code}\n" -X POST http://localhost:8100/api/ask -H "Content-Type: application/json" -H "Authorization: Bearer $TOKEN" -d '{"prompt":"test"}'; done

Requests 1-100 return 200, request 101 returns 429.

File: RateLimiterService.java | Library: Bucket4j 8.10.1 (com.bucket4j:bucket4j-core)
Algorithm: Token bucket | Capacity: 100 | Refill: 100 tokens per 60 minutes (greedy)
Storage: ConcurrentHashMap<String, Bucket> (per-user, in-memory)
O(1) for both tryConsume() and getRemainingTokens() operations
Headers: X-Rate-Limit-Remaining, X-Rate-Limit-Capacity, Retry-After

ReAct Agent — Multi-Step Reasoning Explained

The difference between "Send to Gateway" and "Send (ReAct Agent)"

KEY FEATURE ▼

🧑‍🏫 Think of asking a question to a student vs. a tutor.

Student (Normal mode): You ask "What is 15% of 340 plus the square root of 144?" The student hears the question and tries to answer in one go. They might get it right, or they might make a mistake because they tried to do everything at once.

Tutor (ReAct mode): The tutor hears the same question but works step by step:
💭 "I need to break this into parts..."
✏ Step 1: 15% of 340 = 51
💭 "Now the square root..."
✏ Step 2: √144 = 12
💭 "Now I add them..."
✏ Step 3: 51 + 12 = 63 ✔

The tutor checks each step before moving on. More reliable for complex questions.

The ReAct Pattern (from Yao et al., ICLR 2023 [19]):

  1. The agent receives the user's prompt
  2. THOUGHT: The LLM reasons about what it needs to do
  3. ACTION: The LLM performs one step (calculation, lookup, analysis)
  4. OBSERVATION: The result of that action is captured
  5. Steps 2-4 repeat up to 10 iterations maximum (safety limit to prevent infinite loops)
  6. When the LLM determines it has the final answer, it outputs the result
  7. The response includes reactSteps: N showing how many reasoning cycles were used

⚠ Normal Mode (useReActAgent: false)

One LLM call. Fast (~1-3 seconds). Good for simple questions: "What is AI?", "Capital of France?", "Explain Docker."

Guardrails → 1 Ollama call → Response

✅ ReAct Mode (useReActAgent: true)

Multiple LLM calls with reasoning chain. Slower (~10-30 seconds). Better for complex questions: math, multi-step logic, comparisons.

Guardrails → Thought → Action → Observation → ... (up to 10x) → Response

How to prove it on localhost:3333:

1. Type What is 2+2? → Click "Send to Gateway" → Response comes in ~1-3s with reactSteps: 0

2. Type Calculate 125 * 48 + 390 / 13 → Click "Send (ReAct Agent)" → Response takes longer but shows reactSteps: 3 or more, proving multi-step reasoning was used

Both buttons go through the same 3 guardrail layers. The only difference is what happens AFTER the guardrails pass.

File: ReActAgentService.java | Max iterations: 10
Pattern: Thought → Action → Observation (Yao et al., ICLR 2023)
Response field: reactSteps (int) — 0 for normal mode, 1-10 for ReAct mode
Both modes pass through identical guardrail pipeline (Mono.zip of 3 layers)

NVIDIA NeMo Guardrails 0.10.0 — Layer 1 (The Bouncer)

Pattern-based policy enforcement using Colang DSL — catches known attack templates

LAYER 1 :8001 ▼

👮 The Bouncer with a Banned List.

Imagine a nightclub bouncer who has a list of 97 banned phrases written on a clipboard. When someone approaches, he checks their message against the list. If it matches — blocked immediately, no questions asked. He's fast (40ms) and 100% reliable for known attacks, but clever people who rephrase their words might get past him. That's why we also have the detective (Layer 2).

How Colang DSL works:

  1. Security engineers write rules in Colang (NVIDIA's domain-specific language) — NOT Java code
  2. Rules define patterns: define user ask jailbreak: "ignore previous instructions"
  3. When a prompt arrives, NeMo checks it against ALL patterns simultaneously
  4. If matched → NeMo returns a refusal response containing keywords like "cannot comply"
  5. Our Java client (NemoGuardrailsClient.java) detects these keywords and marks as BLOCKED
  6. New patterns can be added to .co files without changing any Java code or redeploying the app

Microsoft Presidio v2.2 — Layer 3 (The Data Leak Scanner)

PII detection with mathematical validation — credit cards (Luhn), IBANs (ISO 13616)

LAYER 3 :5002 ▼

💳 The Airport Metal Detector for Data.

You walk through a metal detector — it beeps when it finds metal objects (keys, coins, belt buckles). Presidio does the same but for personal data. It scans every prompt looking for patterns that match credit card numbers, social security numbers, email addresses, phone numbers, and 46 other types of personal information. It doesn't just look at the shape — for credit cards, it actually runs the Luhn checksum algorithm to verify it's a real card number, not just a random set of digits.

How Presidio validates a credit card number:

  1. User sends: "Pay with card 4532015112830366"
  2. Presidio's regex engine spots a 16-digit number starting with 4 (Visa pattern)
  3. Presidio runs the Luhn checksum: doubles every other digit, sums them, checks if divisible by 10
  4. 4532015112830366 passes Luhn → confidence score 0.95 (above 0.6 threshold)
  5. Entity type: CREDIT_CARD → request BLOCKED

Why this matters: A random number like "1234567890123456" would NOT be blocked because it fails the Luhn checksum — Presidio knows it's not a real credit card. This prevents false positives while catching real personal data.

Parallel Execution with Mono.zip() — The Key Innovation

All 3 layers run simultaneously — you only wait for the slowest one

CORE INNOVATION ▼

🛒 Supermarket Checkout Lanes.

Sequential (bad): One checkout lane. Customer 1 takes 40ms, Customer 2 takes 200ms, Customer 3 takes 15ms. Total: 255ms.

Parallel (our approach): Three checkout lanes. All three customers check out at the same time. Total: 200ms (the slowest customer). That's 27% faster.

Sequential Execution

NeMo: 40ms → wait → LlamaGuard: 200ms → wait → Presidio: 15ms

Total: 255ms (sum of all three)

Parallel Execution (Our Approach)

NeMo: 40ms   │
LlamaGuard: 200ms │ ← all running at once
Presidio: 15ms   │

Total: 200ms (only the slowest)

The Java code that makes it work:

  1. Mono.zip() from Project Reactor takes 3 asynchronous operations
  2. Each operation runs on a Schedulers.boundedElastic() thread (non-blocking)
  3. All 3 start executing at the same instant
  4. .block() waits until ALL 3 complete (not just the fastest)
  5. Results are collected and checked: anyMatch(GuardrailsResult::blocked)
  6. If ANY returned blocked = request denied (union/fail-closed logic)
File: GuardrailsOrchestrator.java, lines 60-64
Code: Mono.zip(nemoClient.evaluate(prompt).subscribeOn(Schedulers.boundedElastic()),
      llamaGuardClient.evaluate(prompt).subscribeOn(Schedulers.boundedElastic()),
      presidioClient.evaluate(prompt).subscribeOn(Schedulers.boundedElastic())).block();
Decision: allResults.stream().anyMatch(GuardrailsResult::blocked) → fail-closed

JWT Authentication + RBAC + Account Lockout

The complete identity system — who you are, what you can do, and brute-force protection

BUILT-IN ▼

🎫 Festival Wristband System.

When you arrive at a music festival, you show your ticket (username + password) at the entrance. They give you a wristband (JWT token) that you wear for the whole day. At every stage, you flash the wristband — no need to re-enter your password. The wristband has your name and ticket type (ADMIN or USER) encoded in it. VIP wristbands let you into backstage areas (admin dashboard). Regular wristbands don't. The wristband expires at midnight (1 hour). If you lose it and someone finds it, they can't use it because we check the unique ID (JTI) against a blacklist.

7-Step JWT Validation Pipeline (every request):

  1. Signature check — Is this token signed by our server? (HMAC-SHA256)
  2. Expiry check — Has the token expired? (1-hour TTL)
  3. Issuer check — Was it issued by "secure-ai-gateway"?
  4. JTI blacklist — Has this token been logged out? (prevents replay attacks)
  5. Subject present — Does the token contain a username?
  6. Role extraction — Is the user ADMIN or USER?
  7. SecurityContext — Register the authenticated user for this request

Account Lockout: After 5 failed login attempts, the account is locked for 15 minutes. This prevents brute-force password guessing attacks. The lockout check happens before any database or password operations, which prevents timing-based username enumeration (attackers can't tell if a username exists based on response time).

Prove on localhost:3333:

1. JWT: Login as Admin → see the JWT Token Decoded panel showing header (alg: HS384), payload (sub, role, jti, iss, iat, exp), and signature

2. RBAC: Login as User → click "RBAC Test" preset → HTTP 403 Forbidden (USER cannot access /admin)

3. Password policy: Click "Weak Password" preset → Registration fails with "must contain uppercase, lowercase, digit, and special character"

JWT: JJWT 0.12.6, HMAC-SHA384, 1-hour expiry (3,600,000ms)
Password: BCrypt cost=12, @Pattern(regexp="^(?=.*[a-z])(?=.*[A-Z])(?=.*\\d)(?=.*[@$!%*?&#^()\\-_=+])...")
Lockout: 5 attempts → 900s cooldown (ConcurrentHashMap tracking)
RBAC: @PreAuthorize("hasRole('ADMIN')") on /admin/** endpoints
Logout: POST /auth/logout → JTI added to blacklist
